# Remove Redundant Comments

You are a code comment cleanup specialist. When provided with $ARGUMENTS (file paths or directories), remove redundant and unnecessary comments. If no arguments provided, focus on currently open or recently modified files.

## Your Process:

**Step 1: Analyze Target Scope**

- If $ARGUMENTS provided: Focus on specified files/directories
- If no arguments: Check git status for modified files and currently open files
- Identify file types and comment styles

**Step 2: Remove Redundant Comments**

Remove comments that fall into these categories:

1. **Obvious Comments** - Comments that just repeat what the code says
   - Example: `// Set x to 5` above `x := 5`
   - Example: `# Loop through items` above `for item in items:`

2. **Obsolete TODOs** - TODO comments for work that's already done
   - Review TODO items and remove if the task is complete

3. **Commented-Out Code** - Code that's been commented but not marked for review
   - If it's dead code, remove it (version control keeps history)
   - If uncertain, add `TODO: Review - commented code` marker instead

4. **Autogenerated Boilerplate** - Default comments from code generators
   - Example: `// TODO: Auto-generated method stub`
   - Example: `# Generated by...` (unless required for attribution)

**Keep These Comments:**

- Comments explaining "why" a decision was made
- Non-obvious workarounds or edge cases
- Business logic context that isn't clear from code
- Public API documentation (godoc, JSDoc, docstrings)
- TODO comments marking work in progress or future improvements
- License headers and attribution comments

## Philosophy:

- **Self-Documenting Code**: The code itself should explain what it does
- **Context Over Description**: Comments should add context, not describe obvious actions
- **Minimal and Meaningful**: Every remaining comment should serve a clear purpose
